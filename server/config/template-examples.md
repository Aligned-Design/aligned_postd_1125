```

If you're using LAG() to find gaps in user behavior, you're doing it right. But you might be doing it inefficiently.

Here's the thing: Most people calculate time gaps AFTER the query runs. That costs CPU cycles.

Try this instead:

```sql
SELECT
  user_id,
  purchase_date,
  DATEDIFF(day, LAG(purchase_date) OVER (...), purchase_date) AS days_since_last
FROM orders
WHERE DATEDIFF(day, LAG(...), purchase_date) > 30
```

By calculating the gap INSIDE the query, you filter before aggregation. One of our users cut query time by 67%.

## Validation Notes

### Common Issues & Fixes

| Issue | Platform | Fix |
|-------|----------|-----|
| CTA too vague | All | Use action verbs: "Shop", "Register", "Reply", not just "Learn more" |
| Hashtags too generic | IG, X | Research trending + niche hashtags; avoid #followforfollows |
| Medical claims | All | Use "may help", "supports", "promotes"â€”avoid "cures", "treats" |
| No link | Email | Every button/link should have UTM: `?utm_source=email&utm_campaign=launch` |
| Missing urgency | Email | Include deadline, scarcity, or reason to act now |
| Too long headline | All | Test brevity: most engagement comes first 10 words |
